# 历史项目探查检查单 Skill

## 概述

本 Skill 使用经过验证的检查单方法，指导你系统性地探查历史项目。核心原则：**以同类模块代码为唯一真相，拒绝猜测。**

**目标**：为每个类别锁定关键约定与风险点。

## 核心指引

### 渐进式文档策略

**关键要求**：不要生成一个大而全的综合文档。相反：

1. **一次探查一个类别**（按检查单顺序）
2. **完成一个类别后立即生成两个内容**：
   - **通用探查文档**：`01-架构与扩展.md`、`02-环境与构建.md` 等（纯项目结构，不含特定需求）
   - **需求建议追加**：将针对性建议追加到 `08-需求：[需求名称].md`
3. **使用清晰的文件命名**：`01-架构与扩展.md`、`02-环境与构建.md` 等
4. **只有保存当前文件后才进入下一类别**

**文档结构**：
```
openspec/
├── 00-索引.md                    # 项目概览与总结
├── 01-架构与扩展.md              # 通用探查（可复用）
├── 02-环境与构建.md
├── ...
├── 07-风险与债务.md
└── 08-需求：[需求名称].md        # 汇总所有针对性探查与建议
```

这种方法的优势：
- 避免重新生成大文档造成的 token 浪费
- 支持未来开发中选择性引用文件
- **通用探查文档可复用于其他新需求**
- **需求建议集中在一个文档，无需跨文档查找**

### 探查优先级顺序

调查时始终遵循此优先级：
1. **同类模块代码**（首要真相来源）
2. **注释和文档**（仅作次要参考）

### 止损规则

如果任何单点调查超过 30 分钟仍无结果，**立即向人类专家求助**。

## 检查单类别

按顺序执行这些类别，每个类别生成一个文档：

### 1. 架构与扩展锚点 → `01-架构与扩展.md`

**调查内容：**
- 项目整体架构风格（单体/微服务/分层）
- **黄金模板模块**：找出 1 个结构完整、规范清晰的同类模块
  - 完整文件路径
  - 关键文件及其用途
- 可复用基类/服务/组件清单
- 新功能标准接入流程（路由注册、菜单初始化、权限配置等）

**立即输出**：`01-架构与扩展.md`，包含：
- 架构图或描述
- 黄金模块路径和结构
- 可复用组件列表及位置
- 新功能接入分步检查清单

---

### 2. 环境与构建契约 → `02-环境与构建.md`

**调查内容：**
- 环境配置管理方式（环境变量/配置中心）
- 环境差异点
- 构建工具核心配置（路径别名、资源前缀、打包规则）
- 部署产物规范与 CI/CD 关键节点

**立即输出**：`02-环境与构建.md`，包含：
- 环境配置对照表
- 构建配置汇总
- 部署检查清单

---

### 3. 数据层核心约定 → `03-数据层约定.md`

**调查内容：**
- 数据库规范：命名、主键策略、软删除、时区处理
- ORM/数据访问层约定（查询构建、事务、自动字段注入）
- 缓存策略（类型、Key 规范、失效机制）
- 多租户/数据隔离实现层与 tenant_id 来源（如适用）
- 历史数据质量与迁移脚本机制

**立即输出**：`03-数据层约定.md`，包含：
- 数据库约定速查表
- ORM 使用模式与示例
- 缓存规则和 Key 命名模式
- 数据隔离实现细节
- 迁移工作流程

---

### 4. 接口与通信规范 → `04-接口规范.md`

**调查内容：**
- API 设计标准：请求/响应结构、错误码体系、分页参数
- 数据序列化规则：字段命名风格、日期格式、枚举传输方式
- 第三方服务集成方式与认证/重试策略

**立即输出**：`04-接口规范.md`，包含：
- API 请求/响应模板
- 错误码参考表
- 序列化规则与示例
- 第三方集成模式

---

### 5. 安全与权限体系 → `05-安全与权限.md`

**调查内容：**
- 认证流程：令牌存储/刷新/失效处理
- 权限模型：RBAC/ABAC 结构、权限字符串规范、数据权限范围
- 新模块权限初始化步骤与验证方式（遗漏后果？）
- 敏感操作防护机制（CSRF/XSS/输入校验）

**立即输出**：`05-安全与权限.md`，包含：
- 认证流程图
- 权限模型结构
- 权限初始化检查清单
- 安全防护机制

---

### 6. 质量保障基线 → `06-质量保障.md`

**调查内容：**
- 全局错误处理与日志规范（级别、格式、关键埋点）
- 测试策略：单元/集成测试覆盖重点、Mock 方式
- 验证新功能的最小路径（后端接口验证、前端 Mock 方案）
- 监控指标与告警关键点

**立即输出**：`06-质量保障.md`，包含：
- 错误处理模式
- 日志标准与示例
- 测试指南
- 验证检查清单

---

### 7. 风险与债务清单 → `07-风险与债务.md`

**调查内容：**
- 已知技术债务/高风险模块（修改禁忌区）
- 隐性业务规则（文档未载但代码强依赖的逻辑）
- 修改影响范围预判（需回归测试的关联模块）

**立即输出**：`07-风险与债务.md`，包含：
- 高风险模块列表及警告
- 隐性业务规则文档
- 变更影响评估指南

## 执行工作流

当用户请求历史项目探查时：

1. **确认项目访问与需求**：
   - 验证你是否可以访问代码库
   - 确认用户的**具体需求描述**（用于生成针对性建议）
   - 确定需求文档名称：`08-需求：[需求简述].md`

2. **初始化需求文档**：
   - 创建 `08-需求：[需求简述].md`，包含：
     - 需求概述
     - 按阶段分组的建议区域（待填充）

3. **从类别 1 开始**：架构探查

4. **完成每个类别后**（以类别 1 为例）：
   - **生成通用探查文档** `01-架构与扩展.md`：
     - 架构描述、黄金模块路径、可复用组件列表
     - **不包含特定需求的建议**
   - **追加需求建议** 到 `08-需求：[需求简述].md`：
     - 分析该类别对当前需求的影响
     - 提供针对性建议和代码模板
     - 更新需求文档的对应章节
   - 向用户展示摘要
   - 询问："类别 1 完成。是否继续类别 2（环境与构建）？"

5. **对每个类别重复**：探查 → 生成通用文档 → 追加需求建议 → 确认 → 下一个

6. **完成全部 7 个类别后**：
   - 生成 `00-索引.md`，包含：
     - 探查总结
     - 所有类别文件的链接
     - 关键发现速查表（仅通用信息）
     - **指向需求文档的链接**

## 注意事项

1. **通用文档纯净化**：01-07 文档应只包含项目通用信息，避免混合特定需求建议
2. **需求文档聚合**：08 文档是所有针对性建议的单一来源
3. **文档复用**：通用文档可用于多个不同需求的探查，无需重新生成
4. **增量更新**：新需求只需生成新的 08 文档，复用现有通用文档
