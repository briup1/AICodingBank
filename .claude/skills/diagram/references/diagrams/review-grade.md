# 评审场景的图表叠加组件（Review-Grade Add-ons）

> 这不是一种新图表类型，也**不是"评审级图必备的几个要素"**。
>
> 它是**叠加组件清单**——评审/路演场景下，常被叠加在通用图表之上的几个可选组件。
> 用法：先按图表类型规范画基础图，再按本文件**逐组件评估**是否要叠加。
>
> 触发条件：被 writing skill 的 HTML 路演内容调用、用户明确说"画评审版/路演图"、或图表会脱离文档独立使用（截图发群、复制到 PPT）。

---

## 一、设计哲学

评审/路演图常会被独立用于：
- 投屏路演时单图全屏
- 截图发到评审群里
- 打印纸质评审材料
- 复制到 PPT/Keynote 重组

——所以图越能"独立看懂"越好。但**这不等于"必须装满五要素"**。简单的领域关系图就两个椭圆一根线，硬塞标签条和图例反而失焦。

**判断原则**：每个叠加组件都问"图脱离文档独立看，少了它会不会有人读不懂"——会就加，不会就不加。

---

## 二、可叠加组件清单

每个组件按"触发条件 / 实现 / 不要用"三字段描述。

### 二.1 大字标题（22px Bold）

- **触发条件**：图会被投屏（远距离阅读），或被截图独立使用——读者第一眼需要知道"这是什么图"
- **实现**：`title` 字段使用 22px Bold（高于普通图的 16px），格式 `<项目名> · <图类型>`
- **不要用**：图嵌入在文档章节里、章节标题已说清楚——文档配图不需要叠加这个

### 二.2 自解释副标题（14px Regular）

- **触发条件**：单看标题不够，需要一句话讲清楚"这张图想表达什么"
- **实现**：`subtitle` 字段，14px Regular，颜色 `#888`，≤ 30 字
- **不要用**：标题已经自解释（如 "ER 图"）；副标题写不出有意义的内容时不要硬写

### 二.3 顶部标签条（关键词芯片）

- **触发条件**：方案有 3-6 个核心属性需要在第一眼传达（"四层架构 / 模块化 / 事件驱动"），评审/路演场景常用
- **实现**：胶囊形标签横排，N-2 浅底（`#F1F5F9`）+ N-6 文字（`#64748B`），12px Medium
- **不要用**：技术 ER 图、内部数据图——属性概括会显得空洞；超过 6 个时变成"关键词云"反而失焦
- **数量**：3-6 个为宜

### 二.4 节点 emoji icon

- **触发条件**：节点数 ≥ 6 且节点类型多样（用户/系统/数据/外部），icon 帮助快速识读
- **实现**：节点文字左侧加 emoji，与文字基线对齐，16-20px
- **风格规则**：要么全图都用，要么全图都不用——**不要混用**
- **不要用**：
  - ER 图、类图、UML 时序图——数据严谨场景，icon 会破坏专业感
  - 节点少（≤ 5）的简单图——文字已经足够清晰
  - 文字本身就含明显语义（如"订单中心""支付服务"）——icon 显得多余

### 二.5 底部图例条

- **触发条件**：图中用了 ≥ 2 种连线类型，或 ≥ 2 种节点状态色，或 ≥ 2 种容器分组——读者需要"颜色/线型代表什么"的对照
- **实现**：横向排列在图主体下方（间距 ≥ 16px），每项 = 符号（色块/线型）+ 文字标注，12px Regular
- **不要用**：单色单线型图（图例是空话）；图本身就在文档某段"图例说明"段落附近

### 二.6 节点状态色

- **触发条件**：图里有"状态分类"维度——已上线/规划中、待评审/已确认、核心/外围
- **实现**（评审专属配色，对齐 design-system 语义色）：

| 状态 | 填充 | 边框 | 文字 |
|------|------|------|------|
| 已上线 | `#ECFDF5` | `#10B981` | `#065F46` |
| 规划中 | `#FFFBEB` | `#F59E0B` | `#92400E` |
| 待评审 / 核心 | `#EFF6FF` | `#3B82F6` | `#1E40AF` |
| 风险 / 阻塞 | `#FFF1F2` | `#F43F5E` | `#9F1239` |

- **不要用**：节点本身没有"状态"维度时硬塞——颜色会被理解成有信号但实际没含义

### 二.7 多类型连线状态码

- **触发条件**：图中有 ≥ 2 种语义不同的关系（主流程 + 数据流；调用 + 事件；同步 + 异步）
- **实现**：

| 类型 | 样式 | 颜色 |
|------|------|------|
| 主流程 | 2px 实线 | `#3B82F6` |
| 数据流 | 1.5px 实线 | `#10B981` |
| 异步事件 | 1.5px dasharray "6,3" | `#8B5CF6` |
| 安全/校验 | 1.5px dasharray "4,2" | `#F43F5E` |
| 反馈/回调 | 1px dasharray "5,3" | `#94A3B8` |

- **数量上限**：单图 ≤ 3 种连线类型，超过会让评审人迷失
- **不要用**：单一关系类型的图（流程图、ER 图）——所有线都是同一种，不需要状态码

### 二.8 主层 + 侧栏支撑层混合架构（架构图专属）

- **触发条件**：系统有横切的支撑能力（监控/日志/CI/CD/安全）需要展示，且评审人会问"这个系统怎么运维"
- **实现**：见 `architecture.md` 的"主层 + 侧栏支撑层模式"章节
- **不要用**：纯业务流程图、单层服务图——侧栏会显得多余

## 三、组合速查（"内容长什么样 → 加哪些组件"）

| 场景 | 推荐叠加 |
|------|---------|
| 内部技术 ER 图（嵌入文档） | 不叠加任何组件，按 er.md 标准画 |
| 评审用整体架构图（投屏） | 大字标题 + 副标题 + 顶部标签条 + 底部图例 + (可选) 节点 emoji |
| 路演用运行流程图 | 大字标题 + 副标题 + 节点 emoji + 阶段色（节点状态色变体）+ 底部图例 |
| 简单的两实体关系图 | 大字标题（如果独立使用），其他不叠加 |
| 数据中台架构图（含监控/日志侧栏） | 大字标题 + 副标题 + 主层+侧栏混合 + 多类型连线 + 底部图例 |
| 决策树（投屏） | 大字标题 + 副标题 +（不需要 emoji 和标签条） |

## 四、与 SKILL.md 标题策略表的关系

`SKILL.md` 第 2.5 节标题策略表的定位需要重新理解：

**那张表说的是"默认标题行为"——等价于本文件 二.1 + 二.2 两个组件的默认开关**：

| 调用场景 | 默认开关 |
|---------|---------|
| 用户直接请求画图 | 二.1 + 二.2 默认开 |
| deep-research 报告配图 | 二.1 + 二.2 默认关 |
| writing skill 文档配图（嵌入文档） | 二.1 + 二.2 默认关 |
| writing skill 路演 / 评审 HTML 配图 | 二.1 + 二.2 默认开，**其他叠加组件按内容评估** |
| 用户明确说"评审版""路演图" | 进入本文件评估流程，**逐组件判断**是否叠加 |

**关键变化**：被 writing 路演 HTML 调用 ≠ 必须开全部 7 个组件，仍要按内容评估。

## 五、Checklist：评审场景图自检

逐组件评估完成后，过一遍：

- [ ] 大字标题：是否真的会被投屏/独立使用？
- [ ] 副标题：能写出有意义的一句话讲清核心命题吗？写不出就别加
- [ ] 顶部标签条：3-6 个属性能讲清楚方案定位吗？凑数就别加
- [ ] 节点 emoji：是否风格统一？节点数 ≥ 6 且类型多样吗？
- [ ] 底部图例：图里是否真有 ≥ 2 种连线/状态需要说明？
- [ ] 节点状态色：节点有"状态"维度吗？没有就别用
- [ ] 多类型连线：≤ 3 种？语义清晰？
- [ ] 主层+侧栏：架构图且有横切支撑能力时才用

**最关键的一条**：把图脱离文档单独看，能不能讲清楚自己？能 → 已足够；不能 → 看缺哪个组件。

---

## 六、反模式

- ❌ 因为是"评审级"就把 7 个组件全开——评审场景 ≠ 必备五要素
- ❌ 节点 emoji 与不带 emoji 的节点混用——风格不统一
- ❌ 单图超过 3 种连线类型——视觉迷失
- ❌ 顶部标签条 ≥ 7 个关键词——"关键词云"反而失焦
- ❌ ER 图、类图加 emoji——数据严谨场景，icon 破坏专业感
- ❌ 节点本身没有状态维度时硬塞状态色——读者会以为有信号
- ❌ 副标题/标签条凑数硬写——写不出有意义的内容就别加
